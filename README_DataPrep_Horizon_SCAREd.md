# DataPrep Algorithms: Horizon and SCAREd

This repository extends the DataPrep framework with two data quality algorithms, **Horizon** and **SCAREd**, and provides reproducible examples for both data **correction** and **detection** tasks.

The implementation follows a repair-first design:

- `tabular/correction`: generates repaired data tables.
- `tabular/detection`: generates cell-level error masks.
- `examples`: provides runnable scripts, result summaries, changed-cell details, masks, and debug information.

The current implementation focuses on tabular data cleaning experiments and provides an end-to-end workflow on the `hospital_auto` dataset.

---

## Features

- DataPrep-style wrappers for Horizon and SCAREd.
- Separate correction and detection interfaces.
- DataFrame-based inputs and outputs.
- Reproducible command-line examples.
- Automatic result saving, including:
  - repaired data
  - error masks
  - changed/detected cell details
  - metric summaries
  - debug information
- SCAREd repair-range controls through `repair_attrs` and `apply_only_detected`.
- Horizon FD rule discovery and rule-file reuse.

## Algorithm Interfaces

| Algorithm | Correction module | Detection module | Correction output | Detection output |
|---|---|---|---|---|
| Horizon | `tabular/correction/Horizon.py` | `tabular/detection/Horizon.py` | `repaired_df` | `error_mask` |
| SCAREd | `tabular/correction/SCAREd.py` | `tabular/detection/SCAREd.py` | `repaired_df` | `error_mask` |

Both algorithms are implemented as repair-first methods. The correction module produces a repaired table, and the detection module derives an error mask from the cell-wise difference between the repaired table and the original dirty table:

```text
error_mask = repaired_df != dirty_df
```

This design keeps correction and detection behavior consistent while preserving DataPrep-style modularity.

## SCAREd Usage

### Key Parameters

| Parameter | Description |
|---|---|
| `--reliable_attrs` | Attributes used as reliable partition/model inputs, for example `provider_number` |
| `--repair_attrs` | Attributes allowed to be repaired |
| `--apply_only_detected` | Restricts repair to cells marked by the detection mask |
| `--use_perfect_detection_mask` | Builds a controlled evaluation mask from `clean.csv` |
| `--use_clean_in_model` | Passes `clean_df` into the internal workflow for controlled validation |
| `--detection_mask_path` | Loads an external detection mask, such as a mask produced by another detector |

### Why `repair_attrs` Is Used

The original SCAREd workflow relies on the concept of reliable and flexible attributes. In the refactored implementation, `repair_attrs` exposes this repair scope explicitly, allowing users to restrict prediction and repair to a subset of columns.

In the default `hospital_auto` experiment, the following fields are used:

```text
city, county, state, zip, phone, address_1, emergency_service
```

These fields are selected because they are strongly related address, region, and contact attributes. They also correspond well to FD-style dependencies in the dataset, such as address-to-city, address-to-state, and address-to-zip relations. Restricting repairs to these fields helps reduce candidate explosion and over-repair when many flexible attributes are present.

  --use_clean_in_model `

This setting restricts SCAREd to repair only the cells included in the detection mask. It is useful for reducing over-repair and avoiding modifications to cells that were not detected as suspicious.

## Example Results on `hospital_auto`

The following controlled-validation results were obtained on the `hospital_auto` dataset:

| Experiment | Precision | Recall | F1 | Fixed / Total | Changed Cells | New Errors |
|---|---:|---:|---:|---:|---:|---:|
| SCAREd Correction + `repair_attrs` | 79.91% | 36.74% | 50.34% | 187 / 509 | 234 | 45 |
| SCAREd Detection + `repair_attrs` | 80.77% | 37.13% | 50.87% | TP = 189 | Detected = 234 | FP = 45 |
| SCAREd Correction + `apply_only_detected + repair_attrs` | 98.94% | 36.74% | 53.58% | 187 / 509 | 189 | 0 |

Interpretation:

- `repair_attrs` improves the controllability of the repair process by limiting which fields can be modified.
- `apply_only_detected` further reduces over-repair by restricting repairs to detected cells.
- In the controlled setting above, `apply_only_detected + repair_attrs` keeps the number of new errors at zero while repairing 187 true error cells.

## Controlled Validation and Real-World Usage

Some examples use:

```text
--use_perfect_detection_mask
--use_clean_in_model
```

These options are designed for controlled validation, where `clean.csv` is available and can be used to build a perfect detection mask or to analyze repair behavior. These results should be interpreted as a validation of module functionality and repair-range control, not as a fully unsupervised deployment setting.

For real-world use, where ground-truth clean data is unavailable, recommended next steps include:

1. Running SCAREd without `--use_clean_in_model`.
2. Passing an external detection mask through `--detection_mask_path`.
3. Using a detection mask generated by Horizon or another detector.
4. Evaluating `repair_attrs` and `reliable_attrs` on additional datasets.

## Notes

- Horizon uses FD-style constraints and can generate or reuse FD rule files.
- SCAREd is repair-first; detection is derived from changed cells after the repair process.
- Detection-side SCAREd reuses the correction-side candidate-generation logic to avoid divergent implementations.
- `repair_attrs` is an engineering control for explicit repair-scope restriction.
- `debug_info` files are intended to make experiments easier to inspect and reproduce.
